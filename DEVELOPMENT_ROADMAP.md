# 日本动漫观看 APK 开发路线（Flutter + Pencil MCP）

## 1. 项目目标与边界
- 目标：开发 Android APK（后续可扩展 iOS），支持在线观看日本动漫。
- 产品形态：单机个人使用，不提供登录/注册能力。
- 资源来源：通过网络接口获取内容信息与播放地址。
- 指定资源入口：从 `https://mikan.tangbai.cc/` 获取种子/磁力信息后进入播放链路。
- 技术栈：Flutter（前端 + 播放层 + 状态管理）+ 流媒体服务（种子解析、下载调度、转码推流）+ Pencil MCP（UI 设计与设计资产交付）。
- 关键约束：仅接入合法授权的数据源与播放源，避免侵权风险。

## 2. 核心功能规划（MVP -> 增强）
## MVP（第一版必须有）
- 首页推荐（轮播、分区、更新列表）
- 搜索（关键词、筛选、排序）
- 详情页（封面、简介、标签、分集列表）
- 播放页（HLS/MP4 播放、清晰度切换、倍速、进度记忆）
- 收藏与历史记录（本地持久化）
- 基础异常处理（无网、超时、播放失败重试）

## 增强功能（第二阶段）
- 本地个性化（观看偏好、推荐屏蔽词、播放习惯）
- 弹幕/评论（可选）
- 离线缓存（受版权策略限制）
- 推送通知（更新提醒）
- 多语言与主题切换

## 3. 技术架构设计
## 3.1 Flutter 客户端架构
- 架构模式：Clean Architecture（Presentation / Domain / Data）。
- 状态管理：Riverpod（推荐）或 Bloc（二选一，优先 Riverpod）。
- 网络层：Dio + Retrofit（或 Chopper）+ 统一错误拦截。
- 播放器：`video_player` + `better_player`（优先适配 HLS）。
- 本地存储：Hive/Isar（历史、收藏、配置）。
- 路由：GoRouter（支持深链接与路由守卫）。

## 3.2 资源获取与流式播放链路（重点）
- 数据入口：`mikan.tangbai.cc` 页面信息抓取（番剧、分集、发布时间、种子/磁力）。
- 种子处理：解析 `.torrent` / magnet，提交到 BT 下载服务（按需顺序下载，优先视频文件片段）。
- 媒体识别：下载服务输出可播放文件路径，使用 `ffprobe` 检测编码与时长。
- 推流/切片：
  - 方案 A（推荐）：`ffmpeg` 转 HLS（`m3u8 + ts/fmp4`），Flutter 直接拉 HLS 流。
  - 方案 B：`ffmpeg` 推 RTMP 到流媒体网关，再由网关转 HLS 给客户端。
- 播放控制：后端暴露 `play-session`，返回播放 URL、缓冲状态、可选清晰度。
- 缓存策略：热门分集保留切片缓存，冷门资源按 TTL 清理。
- 安全：签名校验、请求频控、错误码标准化（无需用户登录鉴权）。

## 3.3 服务接口（建议）
- `GET /home`
- `GET /search?q=`
- `GET /anime/{id}`
- `POST /play/session`（输入番剧与分集，返回可播放流地址）
- `GET /play/session/{id}/status`（缓冲进度、可播状态）

## 4. Pencil MCP 设计工作流
## 4.1 设计输入
- 产品信息架构：首页、搜索、详情、播放、我的页面流程图。
- 视觉方向：动漫内容型产品，强调封面展示与高信息密度。
- 组件清单：卡片、标签、剧集列表、播放器控制条、弹窗。

## 4.2 设计产出
- 低保真线框（全流程）
- 高保真 UI（暗色/亮色至少一套）
- 设计规范（颜色、字体、间距、圆角、阴影、动效）
- 组件库与状态规范（默认/加载/空态/错误态）

## 4.3 设计到开发交付
- Pencil MCP 输出页面标注与切图规范。
- 开发侧建立 Design Token（`colors.dart`、`spacing.dart`、`typography.dart`）。
- 每个页面先做静态还原，再接数据与交互。

## 5. 分阶段实施计划
## Phase 0：项目初始化（1-2 天）
- 初始化 Flutter 工程结构与模块分层。
- 初始化流媒体服务骨架（抓取、下载、转码、分发目录）。
- 集成基础依赖（网络、状态管理、路由、播放器）。
- 配置 CI（格式检查、静态检查、构建 APK）。

## Phase 1：UI 与框架搭建（3-5 天）
- 基于 Pencil MCP 完成核心页面静态稿落地。
- 建立通用组件库（卡片、骨架屏、错误页、加载页）。
- 完成导航与页面跳转骨架。

## Phase 2：数据接入与业务闭环（5-8 天）
- 打通首页、搜索、详情、播放链路。
- 接入 `mikan` 资源抓取与种子解析。
- 打通“提交播放任务 -> 等待缓冲 -> HLS 播放”完整链路。
- 实现历史记录、收藏、进度记忆。
- 完成统一异常处理与重试策略。

## Phase 3：质量与性能优化（3-5 天）
- 启动速度、列表滚动、图片缓存优化。
- 播放稳定性测试（弱网、切后台、断点恢复）。
- 埋点与日志，定位播放失败原因。

## Phase 4：发布准备（2-3 天）
- 签名打包、混淆配置、隐私政策与用户协议。
- 兼容性测试（不同 Android 版本与机型）。
- 发布首版 APK（内测渠道）。

## 6. 测试与验收标准
- 单元测试：Domain/Data 层核心逻辑覆盖率 >= 60%（MVP 阶段）。
- 集成测试：搜索->详情->播放主链路自动化。
- 验收指标：
  - 首页首屏可见时间 <= 2.5s（中端机）
  - 播放成功率 >= 95%
  - 崩溃率 < 1%

## 7. 风险与应对
- 资源接口不稳定：后端缓存 + 多源兜底 + 熔断降级。
- BT 资源可播延迟：优先顺序下载 + 首段预转码 + 缓冲阈值控制。
- 播放兼容问题：优先 HLS，按机型记录失败画像。
- 版权与合规风险：仅接入授权内容，预留内容下架机制。
- 需求膨胀：严格分 MVP 与增强功能，按里程碑验收。

## 8. 下一步执行清单
- 确认合法资源方与接口协议。
- 用 Pencil MCP 输出首页/详情/播放页第一版高保真稿。
- 初始化 Flutter 工程与流媒体服务骨架。
- 先实现 MVP 主链路：首页 -> 搜索 -> 详情 -> 播放。
